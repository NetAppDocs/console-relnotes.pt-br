=== Resiliência ao Ransomware



==== Suporte do Google Cloud NetApp Volumes (prévia)

O Ransomware Resilience agora oferece suporte a sistemas Google Cloud NetApp Volumes, permitindo que você detecte e responda a ameaças de ransomware no Google Cloud NetApp Volumes com eficiência. Ao realizar a descoberta, o Ransomware Resilience exibe automaticamente as cargas de trabalho dos sistemas Google Cloud NetApp Volumes. Quando você descobre os sistemas, eles aparecem no painel do Ransomware Resilience. O suporte para Google Cloud NetApp Volumes no Ransomware Resilience está atualmente em versão prévia.

Para obter mais informações, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/concept-ransomware-resilience.html["Saiba mais sobre resiliência a ransomware"^]. Para obter informações sobre o escopo do suporte do Google Cloud NetApp Volumes, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/rp-reference-limitations.html["Limitações na resiliência a ransomware"^].



==== Melhorias de restauração limpa

A restauração limpa agora oferece suporte a relatórios. Com os relatórios, você pode revisar e avaliar restaurações limpas, salvando essas informações offline como um arquivo CSV.

A restauração limpa agora também permite interromper a análise. Se você interromper a análise, deverá iniciar a restauração limpa novamente.

Para saber mais sobre relatórios, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/rp-use-reports.html["Baixe relatórios em Resiliência a Ransomware"^]. Para saber mais sobre recuperação, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore-recover.html["Recupere cargas de trabalho com uma restauração limpa"^].



=== Cloud Volumes ONTAP



==== Implantação simplificada do Cloud Volumes ONTAP no Azure

Agora você pode usar a Criação Rápida para implantar Cloud Volumes ONTAP no Azure, tanto para configurações de nó único quanto para configurações de alta disponibilidade (HA). Esse processo simplificado reduz o número de etapas em comparação com o método avançado, define automaticamente os valores padrão em uma única página e minimiza a navegação, tornando a implantação mais rápida e fácil.


NOTE: Se você usar a opção Quick create para implantar um novo sistema Cloud Volumes ONTAP no Azure, o NetApp Console seleciona as credenciais do Azure por padrão. Você pode implantar todos os seus sistemas Cloud Volumes ONTAP usando as credenciais iniciais do Azure. Se quiser usar outra assinatura e/ou credenciais, é necessário alternar para a implantação no modo avançado. Essa limitação será resolvida em uma versão futura.

=== Cloud Volumes ONTAP



==== Faturamento flexível para Cloud Volumes ONTAP e serviços de dados na AWS

NetApp oferece opções de faturamento aprimoradas para suas assinaturas do Cloud Volumes ONTAP na AWS, proporcionando mais controle sobre como você paga pela capacidade do Cloud Volumes ONTAP e pelos serviços de dados da NetApp, como BlueXP backup e recuperação. Anteriormente, todo o uso do Cloud Volumes ONTAP e dos serviços de dados era cobrado em conjunto por meio da listagem NetApp Intelligent Services (NIS) no marketplace da AWS.

Devido a alterações no Programa de Desconto Empresarial (EDP) da AWS, NetApp introduziu uma assinatura independente do Cloud Volumes ONTAP no marketplace da AWS que permite separar a cobrança da capacidade do Cloud Volumes ONTAP dos serviços de dados. Essa separação oferece mais flexibilidade no gerenciamento de seus créditos e custos do EDP. NetApp recomenda o uso da assinatura independente para um melhor gerenciamento de custos do uso do Cloud Volumes ONTAP, enquanto continua usando a assinatura NIS para cobrança de serviços de dados após 31 de julho de 2026, quando os requisitos do EDP entrarão em vigor.

A assinatura independente do Cloud Volumes ONTAP na AWS suporta apenas pacotes de licença Essentials ou Freemium válidos, baseados em capacidade. O pacote Professional não é compatível. Você pode selecionar o método de cobrança que melhor atenda às suas necessidades:

* *Assinatura independente do Cloud Volumes ONTAP (Recomendada)*: Pague pelo uso do Cloud Volumes ONTAP por meio da assinatura independente no AWS Marketplace, enquanto continua a faturar os serviços de dados por meio da assinatura NIS. Esta é a opção recomendada para a maioria dos clientes que possuem assinaturas independentes e NIS associadas à sua conta da AWS no NetApp Console.
* *Assinatura NIS combinada*: Pague pelo Cloud Volumes ONTAP e pelos serviços de dados juntos através da listagem NIS existente. A abordagem combinada será válida para clientes existentes que possuem apenas assinaturas NIS associadas à sua conta AWS no NetApp Console. No entanto, a NetApp recomenda que você defina sua preferência de faturamento para o uso do Cloud Volumes ONTAP e migre para a assinatura independente para um melhor gerenciamento de custos.


Dependendo do seu tipo de assinatura, você terá estes cenários:

.Assinatura avulsa apenas (para novos clientes)
Se você tiver apenas uma assinatura AWS independente associada à sua conta AWS no NetApp Console e nenhuma assinatura NIS, poderá selecionar somente o pacote de licença Essentials durante a implantação. A opção de serviço de dados *backup e recuperação* não estará disponível para você.

.Tanto para assinaturas avulsas quanto para assinaturas NIS (clientes existentes)
Se você tiver assinaturas NIS e independentes associadas à sua conta da AWS e não tiver definido uma preferência de faturamento na guia *Preferências de faturamento* no NetApp Console, o uso do Cloud Volumes ONTAP continuará sendo cobrado pelo NIS por padrão. Esse também é o padrão para qualquer instância do Cloud Volumes ONTAP que execute o pacote Professional, pois a listagem independente não é compatível com o Professional. Você pode definir uma preferência de faturamento e escolher como cobrar cada serviço:

* Defina a assinatura independente como a conta de faturamento preferencial para Cloud Volumes ONTAP (com o pacote Essentials). Se você tiver apenas o pacote Professional, poderá convertê-lo para o pacote Essentials e cobrar o uso do Cloud Volumes ONTAP definindo a preferência de faturamento. Recomendamos essa abordagem porque ela ajuda você a gerenciar seus créditos EDP do Cloud Volumes ONTAP com mais eficiência.
* Utilize a assinatura NIS para faturamento de serviços de dados ou o uso do Cloud Volumes ONTAP por meio do pacote Professional. Durante a implantação, habilite o serviço de dados *Backup e Recuperação* e selecione o pacote de licença Professional para faturar os serviços de dados por meio da assinatura NIS.


.Assinatura NIS apenas (clientes existentes)
Isso se aplica se você já for cliente e possuir assinaturas NIS associadas à sua conta AWS. Você pode continuar faturando tanto o Cloud Volumes ONTAP quanto os serviços de dados juntos, como faz atualmente. No entanto, a NetApp recomenda a migração para o modelo de assinatura independente para um melhor gerenciamento de custos do uso do Cloud Volumes ONTAP:

* Inscreva-se em um novo anúncio independente no portal da AWS.
* Vincule a assinatura independente à sua conta da AWS.
* Configure sua preferência de faturamento para a assinatura independente do Cloud Volumes ONTAP com o pacote Essentials.
* (Se aplicável) Converta seu pacote Professional para um pacote Essentials e defina a preferência de cobrança.
* Continue faturando os serviços de dados através da sua assinatura NIS.



NOTE: Não ocorre medição duplicada quando você usa vários tipos de assinatura ou migra entre listings.

=== Recuperação de desastres



==== Suporte a cargas de trabalho baseadas em Kubernetes (prévia)

NetApp Disaster Recovery agora inclui suporte para cargas de trabalho baseadas em Kubernetes. Com o suporte ao Kubernetes, você pode proteger aplicativos em execução em clusters Kubernetes, incluindo ambientes de OpenShift Virtualization. Disaster Recovery oferece suporte a link:https://docs.netapp.com/us-en/trident/trident-protect/trident-protect-requirements.html["qualquer versão do Kubernetes atualmente suportada pelo Trident Protect"^], garantindo integração perfeita e recursos robustos de recuperação de desastres para aplicativos conteinerizados.

Principais funcionalidades do suporte a cargas de trabalho baseadas em Kubernetes:

* **Ampla compatibilidade**: O Disaster Recovery é compatível com todas as versões do Kubernetes compatíveis com o Trident Protect, incluindo Red Hat OpenShift Virtualization.
* **Proteção centrada na aplicação**: Agora você pode proteger cargas de trabalho do Kubernetes no nível da aplicação, garantindo a recuperação de serviços críticos durante eventos de failover.
* **Orquestração simplificada**: O Disaster Recovery oferece fluxos de trabalho intuitivos para automatizar o backup e a replicação de cargas de trabalho do Kubernetes, aproveitando a mesma tecnologia eficiente SnapMirror usada para ambientes VMware.


O suporte para cargas de trabalho baseadas em Kubernetes está atualmente em versão prévia. Para começar a proteger cargas de trabalho baseadas em Kubernetes, consulte link:https://docs.netapp.com/us-en/data-services-disaster-recovery/get-started/task-kubernetes-requirements.html["Requisitos do Kubernetes para NetApp Disaster Recovery"^].



=== Resiliência ao Ransomware



==== Melhorias de restauração limpa

A restauração limpa agora oferece suporte à recuperação para um local alternativo, permitindo restaurar volumes para diferentes sistemas, VMs de armazenamento e agregados gerenciados pelo mesmo agente do Console. Além disso, a restauração limpa removeu a limitação de sete dias na busca por snapshots viáveis para recuperação. A restauração limpa agora continua procurando por um ponto de recuperação viável entre os snapshots disponíveis, independentemente da distância desde o ataque de ransomware.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-clean-restore-recover.html["Recupere cargas de trabalho com uma restauração limpa"^].



==== Suporte do Google SecOps SIEM

O Ransomware Resilience agora oferece suporte ao gerenciamento de informações e eventos de segurança (SIEM) com o Google SecOps, permitindo que você envie dados automaticamente para o Google SecOps para uma análise e detecção de ameaças simplificadas.

Para obter informações sobre a configuração, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/task-siem.html["Conecte Ransomware Resilience a um SIEM"^].



==== O suporte para Azure NetApp Files agora está geralmente disponível

O Ransomware Resilience agora oferece suporte a sistemas Azure NetApp Files, permitindo que você detecte e responda a ameaças de ransomware no Azure NetApp Files com eficiência. Ao descobrir cargas de trabalho, o Ransomware Resilience agora apresenta Azure NetApp Files e as exibe no painel de proteção. O suporte do Ransomware Resilience para Azure NetApp Files inclui estratégias de detecção e proteção apenas com snapshots. O suporte para Azure NetApp Files já está disponível para o público em geral (GA).

Para obter mais informações, consulte link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/concept-ransomware-resilience.html["Saiba mais sobre resiliência a ransomware"^].



==== Suporte SOAR para bloqueio de usuários

O Ransomware Resilience agora oferece suporte ao bloqueio de usuários individuais com Google SecOps e playbooks do Splunk SOAR. Com essa adição, você pode automatizar o bloqueio de usuários quando o Ransomware Resilience detectar atividades suspeitas de usuários, como uma violação de dados ou exclusão em massa de dados.

link:https://docs.netapp.com/us-en/data-services-ransomware-resilience/reference-soar.html["Integre um playbook SOAR"^].

=== Cloud Volumes ONTAP



==== Capacidade de gerenciar as configurações de proxy do agente do NetApp Console para Cloud Volumes ONTAP

Agora você pode gerenciar as configurações de proxy no agente do NetApp Console para Cloud Volumes ONTAP, mesmo se perder a conexão ou se o proxy estiver configurado incorretamente. Anteriormente, se o agente do Console não conseguisse se conectar ao Cloud Volumes ONTAP por 20 minutos, ele sobrescrevia suas configurações de proxy manuais com as configurações padrão. Isso causava falhas de comunicação, incluindo problemas com mensagens AutoSupport. Para manter suas configurações de proxy para sistemas existentes, execute a seguinte chamada de API:

`PUT /occm/config`

Inclua o seguinte parâmetro no corpo da solicitação:

[source, json]
----
{
  "proxyMode": "No_Overwrites"
}
----
O modo padrão é standard, o que significa que o agente do Console sobrescreve suas configurações de proxy com as configurações padrão caso não consiga se conectar ao Cloud Volumes ONTAP por 20 minutos.

https://docs.netapp.com/us-en/console-automation/cvo/put-occm-config.html["Configurar parâmetros modificáveis do NetApp Console"^]
